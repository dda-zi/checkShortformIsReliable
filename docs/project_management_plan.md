# 프로젝트 관리 계획서

## 1. 프로젝트 개요 및 목표

### 1.1 문서 목적
본 문서는 숏폼 영상의 내용에 대한 신뢰성을 분석하고 사용자에게 판단 근거를 제공하는 소프트웨어 시스템의 개발 및 관리를 위한 계획을 정의한다.  
프로젝트의 개발 절차, 일정, 역할, 품질 관리, 변경 관리, 산출물 관리, 리스크 관리, 개발 환경 및 산출물 작성 계획을 명시하여 프로젝트를 체계적으로 수행하는 것을 목적으로 한다.

### 1.2 프로젝트 개요
본 프로젝트는 사용자가 입력한 숏폼 영상의 링크 또는 콘텐츠 정보를 바탕으로 영상 내 주장 또는 정보의 진위 여부를 분석하고, 그 결과를 신뢰도 점수 및 근거와 함께 제공하는 시스템이다.  
최근 숏폼 플랫폼의 확산으로 짧고 자극적인 정보가 빠르게 소비되고 있으나, 허위 정보나 과장된 정보 또한 쉽게 확산되고 있다. 본 시스템은 사용자가 영상의 내용을 무비판적으로 수용하는 것을 줄이고, 정보의 타당성을 보다 쉽게 검토할 수 있도록 지원하는 것을 목표로 한다.

### 1.3 프로젝트 목표
- 사용자가 숏폼 영상 URL 또는 관련 정보를 입력할 수 있는 기능 제공
- 영상 내 핵심 주장 또는 정보 요소를 추출하는 기능 구현
- 외부 자료 또는 내부 분석 로직을 활용하여 내용의 신뢰도를 분석하는 기능 제공
- 신뢰도 점수와 판단 근거를 사용자에게 시각적으로 제공
- 추후 확장 가능한 구조로 프론트엔드, 백엔드, AI 분석 모듈을 분리하여 설계

### 1.4 범위
#### 포함 범위
- 사용자 입력 인터페이스 구현
- 영상/텍스트 기반 내용 분석 기능
- 신뢰도 평가 결과 출력 기능
- 결과에 대한 근거 정보 제공 기능
- 기본적인 사용자 흐름 설계 및 문서화

#### 제외 범위
- 상용 서비스 수준의 대규모 트래픽 처리
- 완전 자동화된 고정확도 진위 판별 보장
- 모든 숏폼 플랫폼에 대한 완전한 호환성 확보
- 결제 시스템 및 상용화 기능 구현

### 1.5 용어 정의
- **숏폼 영상**: 짧은 길이의 영상 콘텐츠
- **신뢰도 점수**: 분석 결과를 수치화하여 표현한 값
- **근거 정보**: 신뢰도 판단의 이유가 되는 텍스트 또는 참고 데이터
- **프론트엔드**: 사용자와 직접 상호작용하는 화면 영역
- **백엔드**: 데이터 처리, API, 서버 로직 담당 영역
- **AI 모듈**: 텍스트 분석, 주장 추출, 신뢰도 평가 등 분석 기능 담당 영역

### 1.6 참조 문서
- 프로젝트 정의서
- 프로젝트 범위 문서
- 형상관리 계획서
- 요구사항 관련 문서
- GitHub 저장소 문서

## 2. 개발 계획

### 2.1 개발 절차 모형
본 프로젝트는 **점진적 개발이 가능한 반복적/증분적 개발 방식**을 적용한다.  
초기에는 핵심 기능 중심의 최소 기능 제품(MVP)을 우선 개발하고, 이후 분석 정확도 향상 및 사용자 편의 기능을 단계적으로 확장한다.

적용 이유는 다음과 같다.
- 프로젝트 범위가 비교적 넓고 기능 간 의존성이 존재함
- 개인 프로젝트 특성상 일정과 구현 난이도에 따라 우선순위 조정이 필요함
- 초기에 전체 기능을 완성하기보다 핵심 흐름을 우선 구현하는 것이 효율적임

### 2.2 WBS(Work Breakdown Structure)

| WBS ID | 작업 항목 | 세부 내용 |
|---|---|---|
| 1 | 프로젝트 기획 | 주제 선정, 목표 설정, 범위 정의 |
| 1.1 | 문서 작성 | 프로젝트 정의서, 관리 계획서, 형상관리 계획서 작성 |
| 2 | 요구사항 분석 | 기능 요구사항 및 비기능 요구사항 정리 |
| 2.1 | 사용자 흐름 정의 | 입력, 분석, 결과 확인 흐름 설계 |
| 3 | 시스템 설계 | 프론트엔드/백엔드/AI 모듈 구조 설계 |
| 3.1 | 데이터 흐름 설계 | 입력 데이터, 분석 데이터, 결과 데이터 구조 정의 |
| 4 | 프론트엔드 개발 | 입력 화면, 결과 화면, 상태 표시 UI 구현 |
| 5 | 백엔드 개발 | API 설계, 데이터 처리 로직 구현 |
| 6 | AI/분석 모듈 개발 | 주장 추출, 신뢰도 판단 로직 구현 |
| 7 | 테스트 | 기능 테스트, 예외 처리 점검, 통합 테스트 |
| 8 | 문서화 및 정리 | 최종 문서 정리, 발표 자료 작성, 산출물 점검 |

### 2.3 개발 내용
#### 프론트엔드
- 영상 링크 입력 UI
- 분석 요청 버튼 및 로딩 상태 UI
- 분석 결과 및 근거 표시 화면

#### 백엔드
- 사용자 입력 수신 API
- 분석 요청 처리 로직
- 결과 반환 API

#### AI/분석 모듈
- 텍스트 또는 주장 요소 추출
- 신뢰도 판단 기준 설계
- 결과 점수화 및 설명 데이터 생성

본 프로젝트는 개인 프로젝트로 수행되지만, 역할을 다음과 같이 구분하여 관리한다.

| 역할 | 담당자 | 책임 |
|---|---|---|
| PM | 본인 | 일정 수립, 범위 관리, 문서 관리, 전체 진행 점검 |
| 요구사항 분석가 | 본인 | 기능/비기능 요구사항 정의, 사용자 흐름 정리 |

| 테스트 담당자 | 본인 | 기능 테스트, 통합 테스트, 오류 수정 확인 |
| 형상관리 담당자 | 본인 | Git 브랜치 관리, 커밋 기록, 산출물 버전 관리 |

## 4. 품질 관리 및 운영 계획

### 4.1 품질 관리 계획
프로젝트의 품질을 확보하기 위해 다음 항목을 관리한다.
- 요구사항과 구현 결과의 일치 여부 점검
- 화면 및 기능의 정상 동작 여부 확인
- 입력 오류 및 예외 상황 처리 여부 확인
- 분석 결과가 일정한 형식으로 출력되는지 확인
- 문서와 실제 구현 내용의 일치 여부 검토

### 4.2 팀 미팅 계획
개인 프로젝트이므로 별도의 팀 미팅은 없으나, 주 단위로 다음 항목을 스스로 점검한다.
- 이번 주 작업 완료 여부
- 다음 주 작업 목표
- 일정 지연 여부
- 리스크 발생 여부
- 문서 및 산출물 최신화 여부

### 4.3 변경사항 관리
요구사항, 설계, 구현 내용의 변경이 발생할 경우 다음 절차를 따른다.
1. 변경 필요성 확인
2. 영향 범위 검토
3. 관련 문서 수정
4. Git 기록 반영
5. 변경 후 테스트 수행

### 4.4 산출물 관리
모든 산출물은 저장소 내 지정된 폴더에 보관하고 버전을 관리한다.
- 문서 산출물: `docs/`
- 프론트엔드 코드: `frontend/`
- 백엔드 코드: `backend/`
- AI 관련 코드: `ai/`

산출물은 수정 시 커밋 이력을 남기며, 주요 변경 사항은 CHANGELOG 또는 커밋 메시지로 추적한다.

### 4.5 리스크 관리

| 리스크 | 내용 | 대응 방안 |
|---|---|---|
| 일정 지연 | 개인 일정 또는 구현 난이도로 인해 일정이 지연될 수 있음 | 핵심 기능 우선 구현, 부가 기능은 후순위로 조정 |
| 분석 정확도 부족 | 초기 분석 로직의 정확도가 낮을 수 있음 | 규칙 기반 방식으로 먼저 구현 후 점진적 보완 |
| 기술 통합 문제 | 프론트엔드, 백엔드, AI 모듈 간 연동 오류 가능 | 인터페이스를 단순화하고 단계별 연동 테스트 수행 |
| 범위 과다 | 구현 범위가 과도하게 확대될 수 있음 | MVP 기준 유지, 필수 기능 중심으로 범위 제한 |
| 문서-구현 불일치 | 설계 문서와 실제 구현이 달라질 수 있음 | 주차별 문서 업데이트 수행 |

### 5.1 개발 환경
#### 하드웨어 환경
- 개인 노트북 또는 데스크톱 PC
- 인터넷 연결 환경

#### 소프트웨어 환경
- 운영체제: macOS 또는 Windows
- 버전 관리: Git, GitHub
- 프론트엔드: 추후 선정된 UI 개발 환경
- 백엔드: 서버 프레임워크 및 API 개발 환경
- AI 모듈: Python 기반 분석 환경
- 문서 작성: Markdown, PDF 또는 워드 프로세서

- 구현 결과와 문서 내용이 다를 경우 문서를 우선 수정한다.

## 6. 결론
본 프로젝트는 숏폼 영상 정보의 신뢰성을 검토할 수 있는 시스템을 목표로 하며, 개인 프로젝트 특성을 고려하여 반복적/증분적 방식으로 개발한다.  
초기에는 핵심 기능 중심의 MVP를 구현하고, 이후 점진적으로 기능을 확장한다.  
본 관리 계획서를 기준으로 일정, 품질, 문서, 산출물을 체계적으로 관리하여 프로젝트를 수행한다.
